## Lazy (vs eager)

**@ManyToOne(fetch = FetchType.LAZY, optional = false)**

연관된 엔티티를 즉시 조회하지 않고, 실제로 해당 객체를 사용할 때 그때 DB에서 조회하는 방식  
불필요한 조인을 줄여 성능을 최적화하기 위해 사용  
Hibernate는 ManyToOne LAZY라도 프록시를 만들어두고 ID 값은 DB를 안 가고도 읽을 수 있는 경우가 많다.  
(영속성 컨텍스트 1차 캐시)  
ex) getVenue().getId() 정도는 “프록시의 식별자 조회”로 처리돼서 쿼리가 안 나갈 수 있다.



## N+1 예시 상황
**Slot - VenueName 으로 표시해야 하는 상황**  

Slot 목록을 조회 (50개) - **(쿼리 1번)**  
각 Slot마다 slot.getVenue().getName()을 읽음  
Venue 조회가 Slot 개수만큼 추가로 나감 - **(쿼리 50번)**  
=> 총 51번 **(N+1)**

---
lazy 로 불필요한 조회를 막을 수 있지만 위와 같은 상황에서는 여전히 쿼리 폭발

### Fetch Join
```
springframework.data.jpa.repository.Query;

@Query("""
    select s
    from Slot s
    join fetch s.venue v
    order by s.startAt
""")
List<Slot> findAllWithVenueOrderByStartAt();
```
