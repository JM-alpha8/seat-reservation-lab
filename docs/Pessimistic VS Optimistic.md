# 비관락 VS 낙관락

### 핵심 차이 한 줄

비관락(Pessimistic Lock): “충돌 날 것 같으니 미리 잠가서 다른 애를 기다리게 하자”  
낙관락(Optimistic Lock): “대부분 안 겹치니까 일단 진행, 겹치면 나중에 실패/재시도하자”

### 비관락

트랜잭션이 특정 row를 조회할 때 DB row lock을 잡음 (대표적으로 SELECT ... FOR UPDATE)  
다른 트랜잭션이 같은 row를 수정/락 획득하려 하면 대기(block) 하거나 timeout으로 실패  
결과: 충돌을 “발생 자체를 막음(대기시켜서)”

### 낙관락

DB에 락을 미리 잡지 않음  
대신 version 같은 값을 두고, 업데이트 시점에  
“내가 읽었던 버전이 아직 그대로냐?”를 확인  
다르면 OptimisticLockException 발생 → 애플리케이션이 재시도/실패 처리  
결과: 충돌을 “감지해서 실패 처리”

## 시스템에 미치는 영향
### 비관락의 영향

**장점**  
충돌을 거의 확실히 막음 → “중복 예약” 같은 것을 강하게 방지  
재시도 로직이 상대적으로 덜 필요  

**단점/리스크**  
대기(blocking) 증가: 동시 요청이 몰리면 응답이 느려짐(지연 증가)  
데드락 위험: A는 row1→row2 순서로 잠그고, B는 row2→row1 순서로 잠그면 서로 기다리다 데드락  
락 타임아웃/장애 전파: 한 트랜잭션이 오래 잡고 있으면 다른 요청들이 줄줄이 묶임(폭포처럼 느려짐)  
처리량(throughput) 감소: “순차 처리처럼” 되어버리면 초당 처리건수가 떨어짐

비관락은 “안전하지만, 트래픽이 몰릴 때 서비스가 느려지거나 막힐 수 있다”가 핵심.

### 낙관락의 영향

**장점**  
대기가 거의 없음: 락을 안 잡으니 요청이 서로 기다리며 막히는 현상이 적음  
처리량이 높아짐: 충돌이 드문 환경에서는 가장 효율적  

**단점/리스크**  
충돌 시 실패: 동시에 수정하면 한쪽은 예외로 실패 → 사용자에게 “다시 시도”가 필요할 수 있음  
재시도 설계 필수: 자동 재시도(몇 번, 백오프, 어디까지 롤백?)를 잘 설계해야 함  
충돌이 잦으면 오히려 손해: 계속 실패/재시도하면 CPU/DB 업데이트가 낭비되고 응답도 불안정해짐  
낙관락은 “평소엔 빠르지만, 충돌이 많으면 실패와 재시도 비용이 튄다”가 핵심.

## 어떤 상황에 뭐가 더 맞나
**비관락이 어울리는 경우**  
같은 좌석/재고 같은 “핫스팟” 데이터에 동시 접근이 자주 발생  
“절대 중복되면 안 되는” 강한 제약  
실패보다는 “기다려서라도 성공”이 낫다  
예: 좌석 확정(CONFIRMED) 같은 결정적 단계

**낙관락이 어울리는 경우**  
충돌이 드문 편  
사용자 수정(프로필 수정, 게시글 수정)처럼 동시에 같은 레코드를 수정할 확률이 낮음  
“실패하면 다시 시도” UX가 자연스럽게 가능한 경우  
예: 사용자 정보 수정, 설정 저장 등
