## JPA (Java Persistence API)
자바에서 객체(Entity)를 데이터베이스 테이블과 매핑하기 위한 **표준 명세(인터페이스 규격)**  
→ “이렇게 영속성을 구현하자”라는 설계 규칙.

## Hibernate
JPA 명세를 실제로 구현한 **ORM 프레임워크(구현체)**  
→ JPA 규칙을 실제 코드로 동작하게 만드는 엔진.

## @Transactional(readOnly = true)
조회 전용 트랜잭션을 선언하여 데이터 변경을 하지 않음을 명시하는 설정  
Hibernate에서는 변경 감지(Dirty Checking)를 최소화해 조회 성능을 최적화하는 데 사용

## Redis
인메모리 데이터베이스(DBMS) - 데이터 저장소(서버 프로그램)  
Redis는 메모리 기반의 Key-Value 구조 NoSQL 데이터 저장소로, 매우 빠른 읽기·쓰기 속도를 제공  
주로 캐시, 세션 저장소, 실시간 데이터 처리 등에 사용

## Transaction
데이터베이스에서 하나의 논리적 작업 단위로 묶어,  
모두 성공하면 커밋하고 하나라도 실패하면 전체를 되돌리는(롤백하는) 처리 단위.  
즉, “전부 아니면 전무(All or Nothing)”를 보장하는 실행 단위.

## 격리수준(Isolation Level)
동시에 실행되는 트랜잭션들이 서로의 데이터를 어디까지 볼 수 있는지를 정하는 규칙
- READ UNCOMMITTED (거의 안 씀) - 커밋 안 된 데이터도 읽을 수 있음
- READ COMMITTED (가장 많이 씀) - 커밋된 데이터만 읽을 수 있음
- REPEATABLE READ - 같은 트랜잭션 안에서는 같은 데이터를 다시 읽으면 항상 동일 (MySQL InnoDB 기본값)
- SERIALIZABLE - 완전 순차 실행처럼 동작 (가장 느림)

## JWT (JSON Web Token)
JWT는 사용자 인증 정보를 JSON 형태로 담아 서명(Signature) 후 전달하는 토큰 기반 인증 방식  
서버는 세션을 저장하지 않고, 토큰의 서명을 검증하여 사용자를 인증

## Authorization: Bearer <토큰>
HTTP 요청 헤더에 **인증용 토큰(JWT 등)**을 담아 서버에 전달하는 방식  
Bearer는 “이 토큰을 가진 사람이 인증된 사용자다”라는 의미의 인증 스킴

## BCrypt
비밀번호를 안전하게 저장하기 위해 사용하는 단방향 해시 알고리즘  
자동으로 salt를 포함하며, 반복 연산을 통해 무차별 대입 공격에 강하도록 설계됨

## CSRF (Cross-Site Request Forgery)
사용자가 로그인된 상태를 이용해, 공격자가 의도한 요청을 대신 보내게 만드는 공격  
브라우저가 자동으로 쿠키를 포함해 요청을 보내는 특성을 악용  
주로 서버에서 CSRF 토큰 검증으로 방어  

## XSS (Cross-Site Scripting)
웹페이지에 악성 스크립트를 삽입해 사용자의 브라우저에서 실행시키는 공격  
이를 통해 쿠키 탈취, 세션 탈취, 피싱 등이 가능  
출력값 검증 및 이스케이프 처리로 방어  

## 세션 (Session)
사용자 인증 정보를 서버에 저장하는 방식  
클라이언트는 세션 ID만 쿠키로 보관  
서버가 상태를 유지하는 Stateful 인증 방식  

## 쿠키 (Cookie)
브라우저에 저장되는 작은 데이터 조각  
HTTP 요청 시 자동으로 서버에 함께 전송  
세션 ID, 로그인 유지 정보 등에 사용

## 로컬스토리지 (Local Storage)
브라우저에 데이터를 저장하는 공간  
탭이나 브라우저를 닫아도 데이터가 유지  
자동 전송되지 않으며, JavaScript로 직접 접근  

## 세션스토리지 (Session Storage)
브라우저 탭 단위로 데이터를 저장하는 공간  
탭을 닫으면 데이터가 삭제  
로컬스토리지와 달리 세션 범위에서만 유지

## ACID
- **Atomicity (원자성)**  
트랜잭션은 모두 성공하거나 모두 실패해야 한다.  
중간에 일부만 반영되는 일은 없다.  
→ “All or Nothing”


- **Consistency (일관성)**  
트랜잭션 전후에 데이터는 항상 일관된 상태를 유지해야 한다.  
제약조건(무결성 규칙)을 위반하면 안 된다.  


- **Isolation (격리성)**  
동시에 실행되는 트랜잭션은 서로 간섭하지 않아야 한다.  
격리수준에 따라 보이는 데이터 범위가 달라진다.  


- **Durability (지속성)**  
트랜잭션이 커밋되면 그 결과는 영구적으로 저장된다.  
시스템이 장애가 나도 데이터는 유지된다.
